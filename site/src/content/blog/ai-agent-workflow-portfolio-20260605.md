개발 도구는 결과물을 대신 작성하게 두는 것보다 요구사항을 정리하고 변경 범위와 확인 절차를 명확히 하는 데 사용할 때 더 안정적입니다.

## 작업 기준

- 요구사항을 먼저 문서화하고, 빠진 조건을 별도 목록으로 남깁니다.
- 분석, 설계, 구현, 검증 역할을 분리합니다.
- 파일 변경 범위를 명확히 정하고, 변경 후에는 빌드나 테스트로 확인합니다.
- 공개 문서는 담당 역할, 기술 흐름, 확인한 결과 중심으로 작성합니다.

## 가이드로 정리한 내용

SAP 운영개발 환경에서는 ABAP 개발자와 WEB 개발자가 함께 볼 수 있도록 개발 도구 활용 가이드를 구성했습니다. 설치, 로컬 개발 환경, UI5 Controller/View 분석, 브라우저 Network 흐름 확인, 요청 예시를 문서화했습니다.

이 가이드는 운영 서버를 자동 변경하기 위한 것이 아니라, 반복되는 조회와 분석을 더 안전하게 수행하고 결과를 문서화하기 위한 절차입니다.

## 역할 분리형 작업 절차

`multi-agent-work` 공유 패키지는 작업 범위와 최종 판단을 유지하면서 Designer, Developer, Tester, Reviewer, Researcher 역할을 필요에 따라 나누는 방식으로 정리했습니다.

핵심은 시작 조건, 완료 기준, 역할별 확인 결과, 산출물 재검증을 남기는 것입니다.

## 개발 업무와 연결되는 점

요구사항 정리, 변경 범위 관리, 결과 검증은 업무 시스템 운영개발에서도 중요합니다. QA/QC 경력에서 익힌 테스트 관점과 문서화 습관을 개발 작업에도 적용해 변경 결과를 확인하고 기록합니다.
