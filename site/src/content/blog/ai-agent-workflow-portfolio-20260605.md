AI 도구는 결과물을 그대로 대신 만들어주는 도구로 쓰기보다, 작업을 분해하고 검토 루프를 만드는 데 사용할 때 더 안정적입니다.

## 내가 사용하는 기준

- 요구사항을 먼저 문서화하고, 빠진 조건을 별도 목록으로 남깁니다.
- 분석, 설계, 내용 검토, 구현, 검증 역할을 분리합니다.
- 파일 변경 범위를 명확히 정하고, 변경 후에는 빌드나 테스트로 확인합니다.
- 공개 문서에서는 개인정보, 내부 시스템 정보, 고객사 데이터를 제거합니다.

## 사내 가이드로 정리한 내용

SAP 운영개발 환경에서는 ABAP 개발자와 WEB 개발자가 함께 볼 수 있도록 Claude Code 활용 가이드를 구성했습니다. 설치, 로컬 개발 환경, OData 조회 helper, UI5 Controller/View 분석, 브라우저 Network 흐름 확인, 프롬프트 예시를 문서화했습니다.

이 가이드는 운영 서버를 자동 변경하기 위한 것이 아니라, 반복되는 조회와 분석을 더 안전하게 수행하고 결과를 문서화하기 위한 절차입니다.

## 역할 분리형 작업 절차

`multi-agent-work` 공유 패키지는 메인 에이전트가 범위와 최종 판단을 유지하면서 Designer, Developer, Tester, Reviewer, Researcher 역할을 필요에 따라 나누는 방식으로 정리했습니다.

핵심은 여러 AI에게 일을 던지는 것이 아니라, 시작 계약, 완료 기준, 역할별 증거, 산출물 재검증을 남기는 것입니다.

## 포트폴리오에서의 의미

AI를 사용했다는 사실 자체보다 중요한 것은 AI 결과를 어떻게 통제하고 검증했는가입니다. 이 관점은 QA/QC 경력과도 연결됩니다. 테스트 관점, 이슈 추적, 문서화 습관을 AI 작업에도 적용하면 반복 작업을 줄이면서도 결과물의 신뢰도를 유지할 수 있습니다.
